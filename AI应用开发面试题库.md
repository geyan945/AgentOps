# AI应用开发面试题库

副标题：面向大厂 Agent 开发 + 后端开发的综合八股题库  
版本：2026-03 综合重构版  
定位：通用八股主复习材料，不替代 `AgentOps`、`AISmartQA`、`Gated-JSCCFormer` 的项目追问树

## 使用说明

- 这份文档的主轴是 `后端 6 : Agent 4`。也就是说，面试里你既要能讲 `RAG / Agent / Tool / Eval`，也必须能讲 `事务 / 一致性 / Redis / MQ / JVM / MySQL / 接口设计 / 系统治理`。
- 这份文档不是项目深挖树，而是通用题库。项目只做轻度映射，真正的项目追问请回到单独的项目文档。
- 回答时建议固定四步：
  1. 先说定义和本质。
  2. 再说为什么大厂会问这个。
  3. 再说工程要点和高频坑点。
  4. 最后轻点项目，不要每题都硬绑项目。

## 一句话定位

如果面试官问你“你是做 AI 应用，还是做后端”，更稳的回答是：  
我是做 **AI 应用工程 + 后端系统治理** 的交叉型开发，能把 `RAG / Agent / Tool / Eval` 这条 AI 主链，落进 `Java / Spring Boot / MySQL / Redis / RabbitMQ / Python / FastAPI` 的工程骨架里。

## 怎么把通用题落到三个项目

- `AISmartQA`：适合勾连 `RAG / Hybrid Retrieval / ReRank / 引用式回答 / RabbitMQ 异步链 / Redis 锁与限流 / afterCommit / 一致性治理`。
- `AgentOps`：适合勾连 `Agent runtime / LangGraph / checkpoint / approval / MCP / runtime-based Eval / trace / tool governance`。
- `Gated-JSCCFormer`：适合勾连 `模型实验 / 指标分析 / 消融设计 / 研究表达 / 从研究结论抽象工程思路`。
- 如果一道题本质上是通用后端八股，比如 `JVM / MySQL / HTTP / 并发 / Spring`，可以不勾连项目，直接按通用工程答法回答。

## 面试答题总模板

“这个概念本质上是在解决什么问题。大厂问它，通常不是为了背定义，而是看你有没有做过工程取舍和故障收口。工程上我会重点看哪些边界、数据流、失败模式和治理手段。如果要落到项目里，我会轻度映射到最合适的项目场景，但不会强行每题都讲项目故事。”

# 模块1：岗位总纲与回答框架

## 面试题1：大厂 AI 应用开发 / Agent 开发 / 后端开发交叉岗到底在看什么能力？

**标准回答**

它看的不是单一技术点，而是你能不能把模型能力接进一个可治理的系统里。也就是说，既要会 `RAG / Agent / Tool / Eval`，也要会 `事务 / 缓存 / 异步 / 一致性 / 接口 / 可观测性`。

**为什么大厂会问这个**

因为很多候选人会调模型，但不会做系统；也有人后端很强，但讲不清 AI 主链。这个岗位本质上筛的是“交叉型工程能力”。

**工程要点 / 高频坑点**

回答时不要把岗位理解成“会调 API 就够了”，也不要把它讲成“纯后端岗”。更稳的说法是：模型能力是业务能力，后端能力是把业务能力跑稳的骨架。

**可勾连项目**

可总括性地点 `AISmartQA + AgentOps`，强调一个偏检索系统，一个偏 Agent runtime。

## 面试题2：AI 应用开发和普通后端开发的共同点与差异是什么？

**标准回答**

共同点是都要处理接口、数据、缓存、事务、日志、异常和稳定性；差异是 AI 应用多了模型调用、上下文构造、结果波动、评测、Prompt 安全和成本治理。它不是替代传统后端，而是在后端之上多了一层智能链路。

**为什么大厂会问这个**

因为他们想看你有没有“把 AI 特性纳入工程治理”的意识，而不是把 AI 当成一个黑盒外部接口。

**工程要点 / 高频坑点**

不要回答成“AI 应用就是调模型”；也不要把传统后端能力说得无关。AI 系统的稳定性问题，很多时候最后都要回到后端工程能力上。

**可勾连项目**

`AISmartQA` 适合讲后端治理，`AgentOps` 适合讲 AI 运行时治理。

## 面试题3：面试里应该怎么给自己的项目定位，才不会显得像 Demo？

**标准回答**

要先讲系统解决什么问题，再讲主链路，再讲治理闭环。不要上来就说“我用了 LangGraph、Gemini、Milvus”，而要说“我把检索链或 Agent 链落成了可运行、可治理、可评测的系统”。

**为什么大厂会问这个**

因为他们会通过你的开场判断你是“技术名词堆砌型”，还是“工程抽象清晰型”。

**工程要点 / 高频坑点**

最常见的坑是先讲框架名词，再讲功能页面，最后才补系统设计。更稳的顺序永远是：问题、链路、分层、边界、治理、结果。

**可勾连项目**

`AISmartQA` 讲“工程化知识库后端”，`AgentOps` 讲“有 control plane 和 runtime 的 Agent 系统”。

## 面试题4：如果面试官质疑“你这是不是就是调 API 套壳”，怎么答？

**标准回答**

不能急着反驳，应该把复杂性拉回系统。比如说：模型调用只是链路中的一个节点，真正难的是文档接入、检索、工具调用、状态持久化、审批、评测、异步治理和故障收口。

**为什么大厂会问这个**

这是在看你是否真的理解项目价值，还是只是会堆中间件和 AI 名词。

**工程要点 / 高频坑点**

不要回答成“不是，因为我用了很多框架”。要回答成“不是，因为业务复杂性并没有被推给模型，而是被系统显式治理了”。

**可勾连项目**

`AISmartQA` 可讲异步链和 Hybrid 检索，`AgentOps` 可讲 checkpoint、approval、Eval。

## 面试题5：项目题、八股题、系统设计题，面试里应该怎么分配精力？

**标准回答**

项目题决定你有没有真实经历，八股题决定你基础是否扎实，系统设计题决定你有没有抽象能力。三者不能偏废，但对大厂来说，项目和系统设计往往更能拉开差距，八股决定下限。

**为什么大厂会问这个**

他们不一定直接这样问，但整个面试流程就是按这三层在筛人。

**工程要点 / 高频坑点**

复习时不要只背项目追问树，也不要只刷八股。更好的做法是：用八股解释项目里的设计，再用项目证明你不是只会背定义。

**可勾连项目**

这题可不勾连项目，按面试策略回答即可。

## 面试题6：怎么把 AISmartQA、AgentOps、Gated-JSCCFormer 讲成一条能力线？

**标准回答**

可以讲成三层递进：`Gated-JSCCFormer` 证明你有研究和实验能力，`AISmartQA` 证明你能把检索问答做成后端系统，`AgentOps` 证明你开始做 Agent runtime、checkpoint、approval 和 Eval 这类更复杂的 AI 系统工程。

**为什么大厂会问这个**

因为他们想看你的经历是不是彼此割裂，还是已经形成清晰的能力演进路径。

**工程要点 / 高频坑点**

不要把三个项目讲成三个独立孤岛。更稳的讲法是：研究能力负责方法论，RAG 项目负责工程骨架，Agent 项目负责 runtime 和平台化。

**可勾连项目**

三个项目都可点到，但每个只说一句，不展开深追问。

# 模块2：LLM / RAG 基础与检索链

## 面试题7：什么是 LLM 应用？

**标准回答**

LLM 应用不是“调一下模型就结束”，而是把模型接进具体业务流程里，让它承担问答、抽取、分类、写作、工具调用或决策中的一部分能力。它的核心不是模型本身，而是模型与业务系统的结合方式。

**为什么大厂会问这个**

因为很多人会把 LLM 应用理解成简单聊天页面，回答会显得非常浅。

**工程要点 / 高频坑点**

面试里不要只讲模型参数。更重要的是讲数据输入、上下文、工具、输出约束、回退策略和评测方式。

**可勾连项目**

`AISmartQA` 和 `AgentOps` 都可轻度勾连。

## 面试题8：什么是 RAG？核心流程是什么？

**标准回答**

RAG 就是“先检索，再生成”。它先从外部知识库召回相关内容，再把这些内容作为上下文交给大模型生成答案，核心流程通常是：文档接入、解析切分、索引构建、召回、融合、精排、上下文构造、生成和评估。

**为什么大厂会问这个**

因为 RAG 是 AI 应用里最常见的主链路之一，能迅速暴露你对系统完整性的理解深度。

**工程要点 / 高频坑点**

不要只说“就是向量检索加大模型”。更完整的回答要包含切分、融合、精排、引用和评估。

**可勾连项目**

优先勾连 `AISmartQA`。

## 面试题9：RAG 和微调 / SFT 的区别是什么？

**标准回答**

RAG 是把知识放在模型外部，让模型边查边答；SFT 是把行为或部分知识写进模型参数里。前者更新快、可追溯、成本通常更低，后者风格更稳定、延迟更可控，但知识更新成本高。

**为什么大厂会问这个**

因为他们想看你是不是把所有 AI 问题都粗暴归结为“再调个模型”。

**工程要点 / 高频坑点**

不要把 RAG 和 SFT 说成对立关系。更稳的说法是：知识接入优先 RAG，行为稳定和格式约束可以再结合微调。

**可勾连项目**

可轻度提 `AISmartQA` 的知识库场景更适合 RAG。

## 面试题10：一个完整的 RAG 系统至少要拆哪些模块？

**标准回答**

至少要拆成文档接入、解析切分、索引构建、召回、精排、生成、缓存与历史、评估与治理。真正的工程难点不是某一层单点最强，而是这些层如何衔接和闭环。

**为什么大厂会问这个**

因为这题能直接判断你是只做过某个点，还是理解整条链路。

**工程要点 / 高频坑点**

不要漏掉治理层。很多候选人能讲召回和生成，但完全不讲评测、缓存、限流、失败收口。

**可勾连项目**

优先勾连 `AISmartQA`，可顺带提 `AgentOps` 负责评测和运行时治理。

## 面试题11：文档解析为什么不是“把文本读出来”这么简单？

**标准回答**

因为真实文档里有标题、表格、目录、页眉页脚、脚注、格式噪声和结构信息。解析的目标不是“抽出最多文字”，而是保留最有利于检索和回答的语义结构。

**为什么大厂会问这个**

这是在看你有没有真正做过知识接入，而不是只停留在模型层。

**工程要点 / 高频坑点**

粗暴提取全文常常会把噪声送进索引，导致召回错位。解析、清洗、结构保留和切分策略是一套连续设计，不是孤立步骤。

**可勾连项目**

可勾连 `AISmartQA` 的解析链。

## 面试题12：Chunk 常见策略有哪些？怎么选？

**标准回答**

常见有固定长度切分、按结构切分、按语义切分。工程里常见做法不是三选一，而是“先结构、再长度兜底”，在语义完整性和稳定长度之间取平衡。

**为什么大厂会问这个**

因为 chunk 策略直接影响召回、精排和最终回答质量，是 RAG 系统里最容易被忽视但又影响很大的环节。

**工程要点 / 高频坑点**

不要说“256 token 一定最好”这种绝对话。chunk 大小要结合文档类型、topK、模型上下文和误例复盘一起调。

**可勾连项目**

可勾连 `AISmartQA`。

## 面试题13：为什么要做 Overlap？Overlap 多大合适？

**标准回答**

Overlap 是为了防止关键语义被切断，让相邻 chunk 保留必要上下文。它不是越大越好，太大会引入重复召回和噪声，通常会从 chunk 长度的 10% 到 20% 起步调优。

**为什么大厂会问这个**

因为他们想看你是不是知道 chunk 不是纯机械分段，而是信息保持策略。

**工程要点 / 高频坑点**

不要把 Overlap 当成“多存一点数据”。本质上它是在平衡语义连贯性和检索噪声。

**可勾连项目**

可轻度提 `AISmartQA`。

## 面试题14：Embedding 的作用是什么？为什么它是 RAG 的基础？

**标准回答**

Embedding 是把文本映射成可以做语义相似度计算的向量。它的价值不在“把文本变成数字”，而在于让系统能处理同义表达、语义接近和模糊查询。

**为什么大厂会问这个**

这是在确认你理解的不是工具名，而是底层表示方式。

**工程要点 / 高频坑点**

Embedding 做得差，通常最先伤的是召回而不是生成。生成阶段拿不到对的上下文，再强的模型也很难补回来。

**可勾连项目**

可勾连 `AISmartQA` 的向量检索链。

## 面试题15：向量检索什么时候容易失效？

**标准回答**

它在专有名词、编号匹配、精确过滤、短关键词和强词面约束场景下容易失效。因为向量擅长语义相似，不擅长“必须精确命中这个词”。

**为什么大厂会问这个**

因为很多候选人会把向量检索讲得无所不能，这在生产场景里是危险的。

**工程要点 / 高频坑点**

不要把纯向量方案当成通吃方案。企业问答里经常要让 ES/BM25、过滤条件和 ReRank 来补足它的短板。

**可勾连项目**

优先勾连 `AISmartQA` 的 `Milvus + ES`。

## 面试题16：BM25 和 Elasticsearch 在 LLM 时代为什么仍然重要？

**标准回答**

因为关键词匹配、术语命中、过滤、聚合和高亮仍然是很多业务问题的刚需。即使有向量召回，ES 仍然是“找得准”和“加业务约束”的关键部件。

**为什么大厂会问这个**

他们想看你是否真的理解混合检索，而不是一味迷信向量。

**工程要点 / 高频坑点**

不要把 BM25 讲成“老技术已经过时”。在很多编号、术语、错误码、产品名场景里，BM25 仍然非常稳。

**可勾连项目**

优先勾连 `AISmartQA`。

## 面试题17：Hybrid 检索常见怎么做融合？

**标准回答**

常见方法有分数归一化后加权、基于排名的 RRF、规则混合。核心思路不是固定某个权重，而是让语义召回和关键词召回互补，并通过误例复盘不断校准。

**为什么大厂会问这个**

因为真正的工程问题不是“知道 Hybrid”，而是“怎么让 Hybrid 稳定工作”。

**工程要点 / 高频坑点**

不要直接把两路原始分数相加。不同引擎的分数分布往往不可直接比较，通常要先做归一化或改用排名融合。

**可勾连项目**

优先勾连 `AISmartQA` 的 Hybrid 检索。

## 面试题18：为什么召回之后还要做 ReRank？

**标准回答**

因为召回阶段追求的是别漏，精排阶段追求的是排序更准。先把可能相关的候选捞出来，再用更重的模型或更精细的特征做 ReRank，能显著提升送给 LLM 的上下文质量。

**为什么大厂会问这个**

因为这能判断你是否理解“召回”和“精排”是两个不同目标。

**工程要点 / 高频坑点**

不要一开始就想用最强模型全库精排，那样成本和延迟都不可接受。工业上常见做法是召回 TopK，再 ReRank 取前 N。

**可勾连项目**

优先勾连 `AISmartQA` 的 rerank 逻辑。

## 面试题19：怎么判断 ReRank 是否真的有效？

**标准回答**

不能只看“感觉回答更好了”，要看离线指标和失败样本。比如看命中率、引用准确率、上下文相关性、最终回答的 grounded 程度，以及同一评测集上的前后对比。

**为什么大厂会问这个**

因为 AI 系统里最忌讳“拍脑袋优化”，这题在看你有没有评测意识。

**工程要点 / 高频坑点**

不要只比最终答案文本。ReRank 主要作用在“上下文质量”，所以要同时看候选排序和最终回答。

**可勾连项目**

可勾连 `AISmartQA`，也可顺带提 `AgentOps` 的 Eval 思路。

## 面试题20：RAG 系统里的幻觉通常来自哪些环节？

**标准回答**

幻觉通常来自三层：召回错了、上下文构造错了、模型生成越界了。也就是说，幻觉不只是模型问题，很多时候是检索链和上下文工程先出错。

**为什么大厂会问这个**

因为他们想看你是否能把问题拆到系统各层，而不是一股脑甩锅给模型。

**工程要点 / 高频坑点**

排查幻觉时不要只盯 Prompt。先看召回候选，再看精排结果，再看上下文拼装和引用策略，最后才看模型输出约束。

**可勾连项目**

优先勾连 `AISmartQA`。

## 面试题21：怎么做引用式回答和拒答机制？

**标准回答**

引用式回答是让模型基于被检索到的证据回答，并把证据来源一并返回；拒答机制是在证据不足、冲突或低置信度时不硬答。这样做的核心不是“更好看”，而是让答案更可追溯、更可控。

**为什么大厂会问这个**

这是在看你有没有把“可解释性”和“风险边界”纳入系统设计。

**工程要点 / 高频坑点**

不要只返回一段漂亮答案而没有来源。也不要在证据不足时仍强行生成流畅答案。

**可勾连项目**

优先勾连 `AISmartQA` 的 `references`，也可轻点 `AgentOps` 的 citation。

## 面试题22：如果用户反馈“答非所问”，你会怎么排查？

**标准回答**

我会按链路排查：先看 query 是否被正确理解，再看召回候选是否偏了，再看 ReRank、上下文拼装和模型输出。如果是 Agent 场景，还要看 route、tool choice 和 review 结果。

**为什么大厂会问这个**

因为他们在看你有没有系统化定位问题的能力，而不是只会猜。

**工程要点 / 高频坑点**

不要直接说“换个更强模型试试”。先把链路拆开，定位到底是检索错、工具错、上下文错，还是模型约束不够。

**可勾连项目**

`AISmartQA` 可讲检索链，`AgentOps` 可讲 route 和 review。

# 模块3：Agent / LangChain / LangGraph / Runtime

## 面试题23：Agent 和普通 Chatbot 的核心区别是什么？

**标准回答**

普通 Chatbot 更像单轮或多轮对话生成，Agent 则会做规划、调用工具、维护状态、根据观察结果继续决策。前者重点是回答，后者重点是执行和闭环。

**为什么大厂会问这个**

因为很多候选人会把“会聊天”误讲成“会 Agent”。

**工程要点 / 高频坑点**

不要把多轮对话自动等同于 Agent。真正的 Agent 需要明确的动作、观察、状态和停止条件。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题24：什么场景下不建议上 Agent？

**标准回答**

当任务目标非常固定、步骤稳定、风险高、预算紧或根本不需要多步决策时，不建议上 Agent。很多场景用规则流、RAG 或单轮结构化抽取更稳、更便宜。

**为什么大厂会问这个**

因为他们不想要“看见问题就想上 Agent”的候选人。

**工程要点 / 高频坑点**

不要把 Agent 当成默认最优解。越是高风险、强规范、低变化的链路，越要优先考虑规则和确定性流程。

**可勾连项目**

可轻度对比 `AISmartQA` 的 RAG 场景和 `AgentOps` 的多步场景。

## 面试题25：ReAct、Planning、Memory 分别解决什么问题？

**标准回答**

ReAct 解决的是边想边做、边做边观察；Planning 解决的是把任务拆成可执行步骤；Memory 解决的是跨轮或跨阶段保留必要上下文。三者一起，才构成相对完整的 Agent 运行方式。

**为什么大厂会问这个**

因为这能快速看出你对 Agent 基础范式的理解是不是停留在名词层。

**工程要点 / 高频坑点**

不要把 Memory 讲成“保存所有历史消息”。Memory 的重点是保留对后续决策有价值的信息。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题26：Tool Calling 的完整闭环是什么？

**标准回答**

闭环通常是：模型判断是否需要工具、生成工具名和参数、系统做权限和参数校验、执行工具、把结果回填给模型，再决定继续调用还是收尾。关键不是“调工具”，而是“工具调用可治理、可收口”。

**为什么大厂会问这个**

因为 Tool Calling 是 Agent 系统最常见的能力入口。

**工程要点 / 高频坑点**

不要忽略校验、鉴权、超时、重试、幂等和失败回退。很多问题都不出在“模型要不要调用工具”，而出在“系统怎么兜底”。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题27：为什么要拆 Control Plane 和 Runtime？

**标准回答**

因为系统记录源和智能决策层的职责不同。Control Plane 更适合承接鉴权、session、run/step、checkpoint、approval、Eval、工具治理；Runtime 更适合承接状态图执行、模型决策和工具编排。

**为什么大厂会问这个**

这题是在看你有没有平台工程意识，而不是只会在一个进程里堆逻辑。

**工程要点 / 高频坑点**

不要只说“为了解耦”。更关键的收益是：记录源统一、审计统一、治理统一，Runtime 可以迭代但系统事实不散。

**可勾连项目**

优先勾连 `AgentOps` 的 `Java control plane + Python runtime`。

## 面试题28：为什么不是全 Python 或全 Java？

**标准回答**

全 Python 会让 control plane、权限、记录源和业务系统边界更容易混在一起；全 Java 又会让 LangGraph、LangChain、Gemini 这类 runtime 能力落地不自然。分层不是为了炫技，而是为了让每层做自己最擅长的事。

**为什么大厂会问这个**

因为架构分层永远伴随 tradeoff，他们要看你是不是想清楚了。

**工程要点 / 高频坑点**

不要回答成“因为我哪个语言更熟”。面试官要听的是职责边界、演进成本和系统治理收益。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题29：LangChain 和 LangGraph 的边界应该怎么讲？

**标准回答**

更稳的说法是：LangChain 偏模型与工具抽象，LangGraph 偏状态化运行时。前者更像开发效率层，后者更像多步执行、恢复和人工介入的 runtime 层。

**为什么大厂会问这个**

因为很多岗位写这两个名词，但真正筛的是你对系统抽象边界的理解。

**工程要点 / 高频坑点**

不要说成“一个替代另一个”。在复杂 Agent 系统里，它们更像高低层分工。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题30：State / Node / Edge 在 Agent runtime 里分别代表什么？

**标准回答**

State 是当前运行上下文和中间结果，Node 是一步处理逻辑，Edge 是控制流和跳转条件。它们的价值在于让分支、重试、审批和终止条件都能显式表达。

**为什么大厂会问这个**

因为这题能看出你是真的理解 runtime，还是只会画图。

**工程要点 / 高频坑点**

不要把它讲成抽象图论。更重要的是说明：图式建模更适合表达 Agent 的条件跳转和运行态。

**可勾连项目**

优先勾连 `AgentOps` 的 runtime state 和 graph。

## 面试题31：为什么要拆 supervisor 和 reviewer？

**标准回答**

因为一个负责“下一步做什么”，一个负责“当前证据够不够、要不要重规划或审批”。拆开后职责更清晰，trace 更容易读，评测也更容易分维度做。

**为什么大厂会问这个**

因为这题能直接看出你是否理解 Agent 里的“角色分工”，而不是把所有决策塞进一个大 Prompt。

**工程要点 / 高频坑点**

不要只说“拆开更优雅”。更关键的是它让 route、review、replan 和 approval 的边界变得可观测、可评测。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题32：checkpoint 到底解决什么问题？

**标准回答**

checkpoint 解决的是运行过程可恢复，而不是每次失败都从头来。它让系统能在关键节点保存状态，遇到暂停、审批、崩溃或恢复时从已知状态继续执行。

**为什么大厂会问这个**

因为多步 Agent 一旦进入生产场景，恢复能力通常比“跑得出来”更重要。

**工程要点 / 高频坑点**

不要把 checkpoint 讲成简单日志。它必须和 state、一致性、恢复入口、审批恢复以及版本校验绑定。

**可勾连项目**

优先勾连 `AgentOps` 的 `agent_runtime_checkpoint`。

## 面试题33：为什么 resume 要同时校验 resumeToken 和 checkpointVersion？

**标准回答**

因为只校验一个维度不够。resumeToken 用来证明恢复操作对应正确的人和流程，checkpointVersion 用来防 stale resume，避免旧状态覆盖新状态。

**为什么大厂会问这个**

这题能看出你有没有真正考虑恢复一致性，而不是只做了一个“继续执行”按钮。

**工程要点 / 高频坑点**

不要只校验 runId。并发恢复、重复点击、旧审批结果回放，都可能导致状态错乱。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题34：为什么 HITL 不是“模型不够强”的表现？

**标准回答**

human-in-the-loop 首先是风险边界，不是能力妥协。很多高风险操作，比如宽范围 SQL、敏感数据读取、证据冲突，都应该允许人介入确认。

**为什么大厂会问这个**

因为很多人会把人工审批讲成“体验功能”，这会显得系统意识不足。

**工程要点 / 高频坑点**

审批一旦被拒绝，就不能绕过风险继续执行。真正的 HITL 是正式控制流分支，不是页面装饰。

**可勾连项目**

优先勾连 `AgentOps` 的 approval 分支。

## 面试题35：为什么 runtime 要有 max_graph_hops、max_tool_loops、max_replans 这类预算？

**标准回答**

因为 Agent 运行时如果没有明确预算，很容易出现死循环、工具滥用或 reviewer 一直回退。预算的意义是给系统明确停止条件，必要时进入 bounded failure。

**为什么大厂会问这个**

这是在看你有没有资源边界和故障边界意识。

**工程要点 / 高频坑点**

不要把预算讲成“为了保守”。它本质上是成本控制、风控和可预期性的共同约束。

**可勾连项目**

优先勾连 `AgentOps` 的 runtime 预算限制。

## 面试题36：什么是 bounded failure？为什么 Agent 系统需要它？

**标准回答**

bounded failure 指的是：当证据不足、预算耗尽、工具失败或审批拒绝时，系统要在边界内失败，而不是继续乱试或编造答案。它的核心是“可解释地停下”。

**为什么大厂会问这个**

因为真实系统最怕的不是失败，而是不可控地失败。

**工程要点 / 高频坑点**

不要追求“无论如何都给出一个答案”。在高风险场景里，受限失败往往比流畅幻觉更正确。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题37：runtime crash、结构化输出失败、工具失败时该怎么收口？

**标准回答**

要分层收口：结构化输出失败可以重试有限次数，工具失败要记录 observation 和错误类型，runtime crash 要依赖 checkpoint 恢复，超出边界则进入 bounded failure。关键是每层都有清晰的停止和恢复策略。

**为什么大厂会问这个**

因为他们在看你对 failure mode 有没有真正想过。

**工程要点 / 高频坑点**

不要把“catch 一下异常”当成完整方案。更重要的是状态是否保住、trace 是否完整、用户是否拿到可解释结果。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题38：为什么 AgentOps 不是简单的“RAG + tool calling”？

**标准回答**

因为它不只是检索和工具调用，而是有明确的 runtime、状态、checkpoint、approval、review、Eval 和 trace。RAG + tool 解决的是功能，runtime 系统解决的是运行态和治理。

**为什么大厂会问这个**

这其实是在测试你是否理解“能跑”与“可治理”之间的差异。

**工程要点 / 高频坑点**

不要把功能堆叠误说成系统升级。真正的升级是控制流、状态和治理闭环都进入主链。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题39：为什么 Eval 必须跑真实 runtime，而不是旁路脚本？

**标准回答**

因为 Eval 的目标是评估真实系统行为，而不是评估一段理想化离线逻辑。只有跑真实 runtime，route、tool、review、approval、nodePath 和失败模式才会和线上一致。

**为什么大厂会问这个**

因为很多系统的离线评测和线上行为严重脱节，这会让评测失真。

**工程要点 / 高频坑点**

不要只测最终文本。对 Agent 系统来说，route 对不对、工具用得对不对、审批是否触发、是否越界，都很关键。

**可勾连项目**

优先勾连 `AgentOps` 的 runtime-based Eval。

## 面试题40：Trace 最少应该记录哪些内容？

**标准回答**

最少要记录 run、step、node、tool、input/output、observation、状态变化、错误信息、模型版本、prompt 版本和耗时。Agent 系统里 trace 不是“日志增强”，而是排障和评测的基础数据。

**为什么大厂会问这个**

因为没有 trace，多步系统几乎无法做可信排障和复盘。

**工程要点 / 高频坑点**

不要只打最终答案日志。缺少中间决策链路时，任何优化都很难有抓手。

**可勾连项目**

优先勾连 `AgentOps` 的 run/step/graph trace。

# 模块4：Tool Calling / MCP / Context / Memory

## 面试题41：Tool Registry 应该怎么设计？

**标准回答**

Tool Registry 至少要管理工具标识、用途、参数定义、权限要求、超时、重试策略和执行入口。它的作用不是“把函数放到表里”，而是把工具变成可治理资源。

**为什么大厂会问这个**

因为在 Agent 系统里，工具本身就是风险边界和平台资产。

**工程要点 / 高频坑点**

不要只做一个字符串到函数的映射。缺少 schema、权限和治理信息，工具系统很快会失控。

**可勾连项目**

优先勾连 `AgentOps` 的 Tool Registry / MCP。

## 面试题42：为什么工具参数 schema 和参数校验非常关键？

**标准回答**

因为模型生成的参数天然不可信，必须做结构校验、类型校验和业务校验。没有 schema，工具调用很容易从“能跑”变成“误调用、越权调用或危险调用”。

**为什么大厂会问这个**

这是在看你是否真正把模型输出当成不可信输入处理。

**工程要点 / 高频坑点**

不要只做 JSON 反序列化成功就执行。真正风险往往来自字段缺失、枚举非法、范围越界和语义错误。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题43：工具的超时、重试和幂等该怎么设计？

**标准回答**

工具要先按业务类型区分是否可重试、是否需要幂等、以及超时预算是多少。不是所有失败都该重试，也不是所有工具都天然幂等。

**为什么大厂会问这个**

因为 Agent 一旦接真实工具，就会遇到外部系统不稳定和副作用问题。

**工程要点 / 高频坑点**

不要把超时设得无限长，也不要无脑重试副作用型工具。真正稳的做法是把副作用、重试策略和幂等策略绑定设计。

**可勾连项目**

优先勾连 `AgentOps`；也可轻提 `AISmartQA` 的异步幂等。

## 面试题44：为什么工具权限控制必须是最小权限？

**标准回答**

因为模型决策不可完全信任，工具一旦越权，风险会比纯文本生成大得多。最小权限的目标是让每个 run、每类用户、每类工具只拿到完成任务所需的最低能力。

**为什么大厂会问这个**

这题本质上在问你有没有安全和风控意识。

**工程要点 / 高频坑点**

不要把工具权限想成前端按钮控制。真正的权限边界应该在后端和 tool plane。

**可勾连项目**

优先勾连 `AgentOps` 的 tool governance。

## 面试题45：MCP 是什么？它解决了什么问题？

**标准回答**

MCP 可以理解成把工具、资源和交互边界标准化的一种协议思路。它解决的是模型宿主和工具提供方之间的集成一致性问题，让工具接入不再完全依赖私有胶水代码。

**为什么大厂会问这个**

因为很多 Agent 岗已经开始把 MCP 当作工具生态标准化的关键词。

**工程要点 / 高频坑点**

不要把 MCP 讲成“比 Function Calling 更高级”这么空泛。更重要的是标准化接入、治理边界和宿主-工具分层。

**可勾连项目**

优先勾连 `AgentOps` 的 MCP 工具平面。

## 面试题46：MCP 和传统 Function Calling 的区别是什么？

**标准回答**

Function Calling 更像单次调用能力，重点是模型输出一个函数名和参数；MCP 更像工具生态和宿主协作边界，强调资源暴露、能力协商和统一接入方式。前者偏调用，后者偏协议和系统边界。

**为什么大厂会问这个**

因为他们想看你有没有区分“调用机制”和“平台协议”。

**工程要点 / 高频坑点**

不要把两者说成完全替代关系。很多系统里它们解决的问题层级并不一样。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题47：引入 MCP 之后，系统治理上会多出哪些问题？

**标准回答**

会多出工具注册、鉴权、审计、配额、版本兼容、超时、沙箱、输出可信度和故障隔离问题。也就是说，标准化接入降低了集成成本，但提高了治理要求。

**为什么大厂会问这个**

这是在看你是否把协议理解成“接完就完事”，还是理解成“接完才开始治理”。

**工程要点 / 高频坑点**

不要只讲好处不讲代价。任何工具生态化，最后都会把治理复杂度推到平台侧。

**可勾连项目**

优先勾连 `AgentOps` 的 MCP server 和工具权限。

## 面试题48：什么是 Context Engineering？为什么它比“写 Prompt”更重要？

**标准回答**

Context Engineering 是设计模型在某一时刻到底看到什么信息，而不只是写一句系统提示词。它包括历史消息、检索结果、工具返回、记忆、角色说明和输出约束的整体编排。

**为什么大厂会问这个**

因为 AI 应用里很多效果差的问题，本质上都不是 Prompt 文案差，而是上下文工程差。

**工程要点 / 高频坑点**

不要把它理解成“多塞一点上下文”。更重要的是相关性、层次、顺序、噪声控制和长度预算。

**可勾连项目**

`AISmartQA` 可讲上下文构造，`AgentOps` 可讲 runtime state + memory。

## 面试题49：为什么不能把完整会话历史无脑拼进模型？

**标准回答**

因为历史越长，不代表越有用。无脑拼接会带来成本上升、延迟变大、噪声增加和注意力漂移，反而更容易让模型答偏。

**为什么大厂会问这个**

这是在看你有没有上下文预算意识，而不是把大上下文当成万能解。

**工程要点 / 高频坑点**

更稳的做法通常是摘要、筛选、结构化提取和按需召回，而不是整段历史全塞。

**可勾连项目**

优先勾连 `AgentOps` 的 memory / summary 思路。

## 面试题50：Memory 应该存什么，不该存什么？

**标准回答**

应该存对后续决策真正有价值的信息，比如用户偏好、长期目标、身份信息、已确认事实和未完成任务；不该存敏感原文、短期噪声、无关闲聊和没有生命周期管理的内容。

**为什么大厂会问这个**

因为很多候选人会把 memory 讲成“存全部聊天记录”。

**工程要点 / 高频坑点**

Memory 首先是信息治理问题，不只是存储问题。必须考虑时效性、隐私和回收策略。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题51：摘要记忆、向量记忆、结构化记忆怎么取舍？

**标准回答**

摘要记忆适合长对话压缩，向量记忆适合相似回忆，结构化记忆适合事实、偏好和状态。真实系统里通常不是三选一，而是按信息类型组合使用。

**为什么大厂会问这个**

因为他们想看你是否知道 memory 的实现方式和使用场景不是同一回事。

**工程要点 / 高频坑点**

不要把向量记忆当成默认方案。对高价值事实，结构化记忆往往更稳、更可控。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题52：System Prompt、User Prompt、Tool Result 应该怎么分层？

**标准回答**

System Prompt 负责角色和总规则，User Prompt 负责当前任务，Tool Result 负责外部事实输入。分层的目的是避免规则、意图和外部数据混在一起，影响模型判断。

**为什么大厂会问这个**

这题能看出你对提示词工程是不是有结构化思维。

**工程要点 / 高频坑点**

不要把工具返回直接当作高可信自然语言说明。工具结果应当被视为外部输入，并配合约束和过滤使用。

**可勾连项目**

`AISmartQA` 和 `AgentOps` 都可轻度勾连。

## 面试题53：为什么结构化输出在 AI 应用里很重要？

**标准回答**

因为很多下游系统需要稳定字段，而不是一段自由文本。结构化输出能降低解析歧义，提高自动化接入能力，也更便于评测和回归。

**为什么大厂会问这个**

企业场景里，模型输出要接系统，而不只是给人看。

**工程要点 / 高频坑点**

不要只说“方便前端展示”。更关键的是后端可消费、可校验、可测试。

**可勾连项目**

优先勾连 `AgentOps` 的结构化节点输出。

## 面试题54：普通 JSON 输出和 Tool Calling 有什么区别？

**标准回答**

JSON 输出主要解决“按字段返回结果”，Tool Calling 解决的是“触发外部能力并把结果带回闭环”。前者偏格式约束，后者偏系统动作。

**为什么大厂会问这个**

因为很多候选人会把两者混成一回事。

**工程要点 / 高频坑点**

不要把 JSON 输出误认为已经具备工具治理。Tool Calling 还要考虑权限、超时、幂等、失败和结果回填。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题55：为什么 SQL 工具必须做审批和风控？

**标准回答**

因为 SQL 工具一旦接入真实数据，就会涉及高成本查询、宽范围扫描、敏感字段读取和越权风险。相比知识检索，它的副作用和风险边界更高。

**为什么大厂会问这个**

这题是在看你有没有把工具能力按风险分级。

**工程要点 / 高频坑点**

不要只做参数校验就认为够了。还要考虑审批、配额、语句范围、结果裁剪和审计。

**可勾连项目**

优先勾连 `AgentOps` 的 `sql_query` 风控和 approval。

## 面试题56：为什么外部文档和工具返回都应该被视为不可信输入？

**标准回答**

因为它们都可能包含噪声、恶意内容、越权信息或误导模型的指令。对模型来说，外部输入并不会天然分辨“可信事实”和“恶意注入”。

**为什么大厂会问这个**

这题在看你有没有 Prompt Injection 和工具注入防御意识。

**工程要点 / 高频坑点**

不要把“来自自己系统”就等同于可信。真正要做的是分级、过滤、约束和最小权限。

## 面试题57：为什么 Eval 是 AI 系统的核心能力，而不是附属功能？

**标准回答**

因为 AI 系统输出天然有波动，没有 Eval 就无法知道优化到底是真的提升了，还是只是换了一种错误。Eval 不是锦上添花，而是迭代闭环的基础。

**为什么大厂会问这个**

很多团队做 AI 应用最先踩的坑，就是改了很多东西却没有可比性。

**工程要点 / 高频坑点**

不要把 Eval 理解成“人工随便测几个问题”。真正可用的 Eval 要有样本、指标、版本和回归流程。

**可勾连项目**

优先勾连 `AgentOps`，也可轻提 `AISmartQA` 的检索效果评估。

## 面试题58：离线评测、线上指标、人工评审分别解决什么问题？

**标准回答**

离线评测解决可重复比较，线上指标解决真实使用行为观察，人工评审解决复杂质量判断。三者互补，谁都不能完全替代谁。

**为什么大厂会问这个**

因为成熟团队不会只靠某一类指标做迭代决策。

**工程要点 / 高频坑点**

不要把线上点击率或满意度当成全部真相，也不要只信离线分数。AI 系统常常需要多维度一起看。

**可勾连项目**

优先勾连 `AgentOps` 的 Eval 思路。

## 面试题59：评测集应该怎么构建，才真的能指导迭代？

**标准回答**

评测集要覆盖主场景、边界场景、失败高发场景和风险场景，并且要能稳定复现。它不是题库越多越好，而是分布、标签和版本管理要合理。

**为什么大厂会问这个**

这是在看你有没有把评测当数据资产来维护。

**工程要点 / 高频坑点**

不要只挑“最容易答对”的样本。更有价值的是高频问题、难例和真正影响用户体验的样本。

**可勾连项目**

优先勾连 `AgentOps`，也可轻提 `Gated-JSCCFormer` 的实验设计意识。

## 面试题60：为什么 route、grounding、citation、approval 要拆成多个评测维度？

**标准回答**

因为 Agent 系统不是只有最终答案文本。route 决定有没有走对分支，grounding 决定有没有基于证据，citation 决定可追溯性，approval 决定风险边界是否被正确触发。

**为什么大厂会问这个**

这题在看你是否理解多步系统不能只看单一分数。

**工程要点 / 高频坑点**

不要只给一个总分。总分适合汇总，但问题定位必须拆回各个维度。

**可勾连项目**

优先勾连 `AgentOps` 的 Eval 指标设计。

## 面试题61：AI 系统里的 A/B 测试应该怎么做？

**标准回答**

A/B 的核心不是“同时跑两个版本”，而是控制变量、选对指标、保证流量分桶合理，并把 Prompt、模型、工具版本纳入对比对象。它的目的是验证改动是否在真实流量里成立。

**为什么大厂会问这个**

因为很多 AI 优化在离线看起来有效，到了线上并不成立。

**工程要点 / 高频坑点**

不要把多个变量一起改，否则无法归因。AI 系统里尤其要注意模型版本、上下文构造和工具行为的联动影响。

**可勾连项目**

可不强绑项目，按通用 AI 系统答法即可。

## 面试题62：失败样本分析应该怎么做？

**标准回答**

先按失败类型归类，再看链路在哪一层出错，然后针对高频失败制定改动计划。比起“翻几个案例看一眼”，更重要的是形成可复用的错误分类和回归样本。

**为什么大厂会问这个**

因为真正能推动系统进步的，不是平均表现，而是你怎么处理失败样本。

**工程要点 / 高频坑点**

不要把失败样本分析做成随手感想。至少要有分类维度、案例沉淀和回归机制。

**可勾连项目**

`AgentOps` 可讲 Eval 失败样本，`AISmartQA` 可讲检索误例。

## 面试题63：AI 应用怎么做成本治理？

**标准回答**

成本治理不是只换便宜模型，而是从路由、缓存、上下文裁剪、工具预算、模型分层、异步化和拒答策略多个层面一起做。核心是把“高成本调用”留给真正值得的请求。

**为什么大厂会问这个**

因为 AI 应用的成本很容易在模型、检索和工具层同时膨胀。

**工程要点 / 高频坑点**

不要只盯单次调用价格。更要看端到端链路成本、失败重试成本和长上下文成本。

**可勾连项目**

`AISmartQA` 可讲检索与缓存，`AgentOps` 可讲 runtime budget。

## 面试题64：一个 AI 应用的延迟应该怎么拆？

**标准回答**

通常要拆成请求预处理、检索、工具调用、模型推理、后处理和前端展示多个阶段。延迟优化的前提是知道时间到底花在哪一层，而不是笼统说“模型慢”。

**为什么大厂会问这个**

因为很多优化都失败在“没量化基线”。

**工程要点 / 高频坑点**

不要只看平均耗时。P95、P99 和异常请求的长尾才是真实用户体验的关键。

**可勾连项目**

`AISmartQA` 可讲检索延迟，`AgentOps` 可讲多步 runtime 延迟。

## 面试题65：AI 系统里可观测性最少应该看什么？

**标准回答**

最少要看日志、TraceId、链路耗时、错误率、模型版本、工具调用记录、缓存命中、队列积压和关键业务指标。对多步系统来说，还要能看到 route、nodePath 和 checkpoint 状态。

**为什么大厂会问这个**

AI 系统链路长、随机性强，没有可观测性就无法排障和优化。

**工程要点 / 高频坑点**

不要把可观测性理解成只打应用日志。真正有效的是把链路、指标、事件和版本一起串起来。

**可勾连项目**

`AISmartQA` 可讲任务治理，`AgentOps` 可讲 trace / graph / Eval。

## 面试题66：什么是 Prompt Injection？为什么 RAG 和 Agent 更容易中招？

**标准回答**

Prompt Injection 是外部输入通过伪装成指令来干扰模型行为。RAG 和 Agent 更容易中招，是因为它们会摄入外部文档、工具返回和网页内容，这些都可能把恶意指令带进上下文。

**为什么大厂会问这个**

因为 AI 系统的安全风险很多不在传统接口层，而在模型输入层。

**工程要点 / 高频坑点**

不要以为“文档是内部的”就安全。风险控制要靠输入分层、过滤、最小权限、审批和输出约束。

**可勾连项目**

`AISmartQA` 可讲文档注入，`AgentOps` 可讲工具结果注入。

## 面试题67：为什么要做日志脱敏、权限隔离和敏感数据控制？

**标准回答**

因为 AI 系统会集中处理大量原始文本、工具返回和用户数据，天然更容易把敏感信息泄露到日志、缓存或评测集中。权限隔离和脱敏是基本工程要求，不是上线后再补的细节。

**为什么大厂会问这个**

这是在看你对数据安全是不是有底线意识。

**工程要点 / 高频坑点**

不要只盯数据库权限。日志、评测样本、缓存和 trace 都可能成为泄露面。

**可勾连项目**

## 面试题68：为什么 AI 系统必须有降级、兜底和拒答策略？

**标准回答**

因为模型、工具、检索和外部服务都可能失败。降级和拒答的目标不是“体验差一点”，而是让系统在失败时仍然可解释、可预期、可恢复。

**为什么大厂会问这个**

真实系统不可能只讨论理想路径，这题在看你有没有最坏情况意识。

**工程要点 / 高频坑点**

不要追求“任何时候都给完整答案”。很多场景下，受限输出和明确拒答更专业。

**可勾连项目**

`AISmartQA` 可讲检索失败兜底，`AgentOps` 可讲 bounded failure。

## 面试题69：配置管理和密钥管理为什么在 AI 工程里尤其重要？

**标准回答**

因为 AI 系统通常依赖多个模型服务、向量库、检索引擎和消息系统，配置复杂且环境差异大；同时 API Key、数据库密码和内部 token 都属于高敏感信息。配置和密钥管理做不好，系统既不安全，也不稳定。

**为什么大厂会问这个**

因为很多候选人项目看起来能跑，但一问环境管理就很薄弱。

**工程要点 / 高频坑点**

不要把密钥写死在代码和配置文件里，也不要把环境差异藏在本地 IDE。更稳的做法是分环境配置、环境变量和权限控制。

**可勾连项目**

至少要覆盖接口回归、评测集回归、关键工具回归、权限回归和失败路径回归。因为 Prompt、模型、上下文和工具任何一个变化，都可能带来行为变化。

AI 系统的变更面比普通接口更宽，上线风险也更隐蔽。

**工程要点 / 高频坑点**

# 模块6：Java / JVM / Spring Boot

## 面试题71：Java 集合体系里，List、Set、Map 最该怎么讲？

**标准回答**

List 强调有序可重复，Set 强调去重，Map 强调键值映射。面试里更重要的不是背接口继承图，而是说清楚不同场景下为什么选它们。

这是后端基础能力的下限题，但也能延伸到性能和并发。

**工程要点 / 高频坑点**

不要停留在“List 能重复、Set 不能重复”这种层次。更好的回答会顺带补实现类、时间复杂度和使用场景。

**可勾连项目**

可不强绑项目，按通用 Java 基础回答即可。

## 面试题72：HashMap 为什么线程不安全？什么时候该用 ConcurrentHashMap？

**标准回答**

HashMap 在并发写场景下会出现数据竞争、覆盖和结构不一致问题，所以线程不安全。ConcurrentHashMap 适合高并发读写但又不想用全局锁串行化的场景。

**为什么大厂会问这个**

这题是经典并发基础，同时能看出你是不是会把容器和线程安全问题联系起来。

**工程要点 / 高频坑点**

不要把 ConcurrentHashMap 理解成“任何时候都能替代 HashMap”。单线程或只读场景下，HashMap 更简单。

**可勾连项目**

可不强绑项目。

## 面试题73：ArrayList 和 LinkedList 怎么选？

**标准回答**

大多数工程场景优先选 ArrayList，因为它的随机访问友好、内存局部性更好。LinkedList 只有在你真的有大量中间插入删除、且不依赖随机访问时才可能成立。

**为什么大厂会问这个**

因为这是典型的“背概念很容易，做工程很少选错”的题。

**工程要点 / 高频坑点**

不要机械背“LinkedList 插入删除快”。现代工程里，CPU 缓存和访问模式常常让 ArrayList 更实用。

**可勾连项目**

可不强绑项目。

## 面试题74：为什么 equals 和 hashCode 经常要求成对重写？

**标准回答**

因为哈希容器先靠 hashCode 定桶，再靠 equals 判断相等。如果只重写一个，Map/Set 的行为就会不一致，可能出现逻辑重复或查找失败。

**为什么大厂会问这个**

这是 Java 语义一致性的经典题，能快速看出基础是否扎实。

**工程要点 / 高频坑点**

不要只说“规范要求这样做”。要讲清楚这和 HashMap、HashSet 的行为契约直接相关。

**可勾连项目**

可不强绑项目。

## 面试题75：String、StringBuilder、StringBuffer 的区别是什么？

**标准回答**

String 不可变，适合只读和共享；StringBuilder 可变且非线程安全，适合单线程拼接；StringBuffer 可变且线程安全，但通常因为同步开销较大而较少作为默认选择。

**为什么大厂会问这个**

这题考基础，也能看出你是否理解不可变对象和线程安全的 tradeoff。

**工程要点 / 高频坑点**

不要把 String 不可变只说成“不能修改”。更关键的是线程安全、共享和哈希稳定性收益。

**可勾连项目**

可不强绑项目。

## 面试题76：JVM 内存模型最该讲什么？

**标准回答**

更稳的答法是：JVM 里要区分运行时内存区域和并发语义层面的内存模型。面试里常说的 JMM 更关注可见性、有序性和原子性，而不是单纯的堆栈划分。

**为什么大厂会问这个**

因为很多候选人会把“JMM”和“堆栈方法区”混为一谈。

**工程要点 / 高频坑点**

如果题目不明确，先澄清自己回答的是并发语义层面的 JMM。然后再补 volatile、synchronized 和 happens-before。

## 面试题77：堆、栈、方法区分别放什么？

**标准回答**

堆主要放对象实例，栈主要放栈帧和局部变量，方法区主要放类元数据、常量和静态信息。面试里更重要的是说清楚它们和对象生命周期、GC、线程隔离之间的关系。

**为什么大厂会问这个**

这是 JVM 基础题，也经常作为 GC 和 OOM 的前置题。

**工程要点 / 高频坑点**

不要死记版本细节而忽略主线。主线永远是对象在哪、谁共享、谁回收、谁按线程隔离。

**可勾连项目**

可不强绑项目。

## 面试题78：GC 常见算法和收集器该怎么答？

**标准回答**

算法层面通常说标记-清除、复制、标记-整理；收集器层面常见有 Serial、Parallel、CMS、G1。更稳的答法不是背名字，而是说明它们在吞吐量、停顿时间和复杂度上的取舍。

**为什么大厂会问这个**

因为 GC 会直接影响后端延迟和稳定性。

**工程要点 / 高频坑点**

不要把所有收集器讲成“越新越好”。要结合业务特征，比如更重吞吐量还是更重低停顿。

**可勾连项目**

可不强绑项目。

## 面试题79：类加载过程是什么？

**标准回答**

一般会说：加载、验证、准备、解析、初始化。更关键的是理解什么时候会触发类初始化，以及静态变量和静态代码块的执行时机。

**为什么大厂会问这个**

这是 JVM 基础题，也常被用来考察你对框架启动过程的理解。

**工程要点 / 高频坑点**

不要只机械背五个阶段。更有价值的是能解释一个类为什么在某个时刻才真正初始化。

**可勾连项目**

可不强绑项目。

## 面试题80：双亲委派为什么存在？

**标准回答**

双亲委派的核心目的是避免类的重复加载和核心类被随意篡改。它让类加载有稳定的查找顺序，从而保证 Java 核心类库的一致性和安全性。

**为什么大厂会问这个**

这是理解类加载机制和框架隔离的基础题。

**工程要点 / 高频坑点**

不要只说“先问父加载器”。更要说明它解决了什么问题，以及为什么某些容器会打破部分委派。

**可勾连项目**

可不强绑项目。

## 面试题81：线程池七参数最该怎么答？

**标准回答**

核心线程数、最大线程数、空闲存活时间、时间单位、阻塞队列、线程工厂、拒绝策略。更关键的是解释这些参数一起决定了线程池的容量模型和背压方式。

**为什么大厂会问这个**

线程池是后端系统最常见的资源治理入口之一。

**工程要点 / 高频坑点**

不要把七参数背成口诀就结束。更重要的是说清楚任务积压时到底先扩线程、还是先排队、还是触发拒绝。

**可勾连项目**

可不强绑项目。

## 面试题82：为什么不建议直接用 Executors 默认工厂方法？

**标准回答**

因为很多默认工厂会隐藏危险配置，比如无界队列或无限线程数，在高压场景下容易导致内存暴涨或线程失控。手动创建 ThreadPoolExecutor 更容易做到容量可控。

**为什么大厂会问这个**

这是在看你有没有资源边界意识。

**工程要点 / 高频坑点**

不要回答成“因为面试喜欢这么问”。本质是默认配置对生产系统不透明、不安全。

**可勾连项目**

可不强绑项目。

## 面试题83：synchronized、ReentrantLock、CAS 分别适合什么场景？

**标准回答**

synchronized 适合简单互斥场景，ReentrantLock 适合需要可中断、可超时、公平锁或条件变量的场景，CAS 适合短临界区和无锁原子更新场景。它们不是谁全面替代谁，而是不同层次的并发工具。

**为什么大厂会问这个**

因为这题能看出你是不是把并发工具当成“越高级越好”。

**工程要点 / 高频坑点**

不要把 CAS 讲成没有成本。自旋失败和 ABA 等问题都要考虑。

**可勾连项目**

可不强绑项目。

## 面试题84：volatile 解决什么问题，解决不了什么问题？

**标准回答**

volatile 主要解决可见性和有序性问题，不能单独保证复合操作的原子性。也就是说，它适合状态标记和配置刷新，不适合替代锁来保护复杂共享状态。

**为什么大厂会问这个**

这是并发语义基础题，经常用来筛掉概念不清的人。

**工程要点 / 高频坑点**

不要说“volatile 保证线程安全”。这会被立刻追问到自增、计数器等反例。

**可勾连项目**

可不强绑项目。

## 面试题85：ThreadLocal 有什么价值，也有什么风险？

**标准回答**

ThreadLocal 适合在线程上下文里传递用户信息、TraceId 或事务相关上下文，避免层层传参。风险在于线程池复用时容易造成数据残留和内存泄露。

**为什么大厂会问这个**

因为很多后端框架和链路追踪都会用到它。

**工程要点 / 高频坑点**

用完要及时清理，尤其在线程池和长生命周期线程里。否则脏数据和泄露问题都很难排查。

**可勾连项目**

可不强绑项目。

## 面试题86：IOC 和 AOP 在 Spring 里最该怎么讲？

**标准回答**

IOC 是把对象创建和依赖管理交给容器，AOP 是把横切逻辑从业务代码里抽出来。前者解决对象组织问题，后者解决事务、日志、权限等横切关注点问题。

**为什么大厂会问这个**

这是理解 Spring 核心思想的基础题。

**工程要点 / 高频坑点**

不要只背全称。更有价值的是讲它们为什么让业务代码更聚焦，也带来了哪些代理和生命周期问题。

**可勾连项目**

可不强绑项目。

## 面试题87：Spring Boot 自动装配和启动流程，面试里该讲到什么深度？

**标准回答**

一般讲到 `@SpringBootApplication`、自动配置扫描、条件装配和容器启动流程就够了。大厂面试更看重你能否把自动装配和业务行为联系起来，而不是背源码细节。

**为什么大厂会问这个**

这是在看你对框架“为什么能工作”有没有基本理解。

**工程要点 / 高频坑点**

不要为了显得深入而背很多类名，却说不清条件装配解决了什么问题。

**可勾连项目**

可不强绑项目。

## 面试题88：@Transactional 什么时候该加，什么时候不能乱加？

**标准回答**

当一组数据库操作必须作为一个原子单元成功或失败时应该加事务；但事务边界不应把外部系统副作用、长耗时操作和不必要的远程调用包进去。事务是为了收紧核心状态边界，不是为了把所有事都包起来。

**为什么大厂会问这个**

因为事务是后端系统里最常见也最容易被误用的能力。

**工程要点 / 高频坑点**

不要在事务里做大范围外部调用，也不要以为加了事务就自动解决分布式一致性。外部副作用常常要 afterCommit 或异步补偿。

**可勾连项目**

优先勾连 `AISmartQA` 的 afterCommit 场景。

## 面试题89：Spring Bean 生命周期最该怎么讲？

**标准回答**

通常说实例化、依赖注入、初始化、使用、销毁即可。更有价值的是知道 BeanPostProcessor、初始化回调和销毁回调分别在什么时机介入。

**为什么大厂会问这个**

这题能看出你是否理解 Spring 容器不是黑盒。

**工程要点 / 高频坑点**

不要把生命周期背成细碎流程图却讲不出业务意义。面试里更重要的是“哪些扩展点能做什么”。

**可勾连项目**

可不强绑项目。

## 面试题90：Controller、Service、Repository 的边界应该怎么划？

**标准回答**

Controller 负责协议和参数层，Service 负责业务编排和事务边界，Repository 负责数据访问。分层的目的不是形式统一，而是让协议、业务和存储边界清晰。

**为什么大厂会问这个**

这是在看你有没有基本的后端设计意识。

**工程要点 / 高频坑点**

不要把业务逻辑堆到 Controller，也不要让 Repository 承担业务编排。分层失真后，测试和演进都会变难。

**可勾连项目**

`AISmartQA` 和 `AgentOps` 都可轻度勾连。

# 模块7：MySQL / 事务 / 索引 / 锁

## 面试题91：MySQL 索引为什么能加快查询？

**标准回答**

因为索引把全表扫描变成了更高效的有序查找或范围查找，本质上是用空间换时间。面试里更稳的说法是：索引减少了需要访问的数据量和随机扫描成本。

**为什么大厂会问这个**

索引是后端性能基础题，也是很多系统优化的起点。

**工程要点 / 高频坑点**

不要只说“有 B+Tree 所以快”。还要说清楚减少扫描、支持排序和范围查询这些收益。

**可勾连项目**

可不强绑项目。

## 面试题92：聚簇索引和二级索引有什么区别？

**标准回答**

聚簇索引的数据和主键顺序绑定，叶子节点直接存整行数据；二级索引叶子节点通常存索引列和主键值，需要时再回主键索引取完整行。二者的差异直接决定了回表成本。

**为什么大厂会问这个**

因为理解这个，才能真正理解查询为什么快或慢。

**工程要点 / 高频坑点**

不要只背定义。更关键的是把它和回表、覆盖索引、主键设计联系起来。

**可勾连项目**

可不强绑项目。

## 面试题93：什么是回表？什么是覆盖索引？

**标准回答**

回表是指通过二级索引找到主键后，再去聚簇索引取完整行；覆盖索引是查询所需字段都在索引里，不需要再回表。覆盖索引的价值在于减少随机 IO 和访问成本。

**为什么大厂会问这个**

这是索引优化里很实用的高频题。

**工程要点 / 高频坑点**

不要把覆盖索引理解成“索引列越多越好”。索引也有维护成本和空间成本。

**可勾连项目**

可不强绑项目。

## 面试题94：最左前缀原则是什么？

**标准回答**

联合索引从左到右生效，如果查询条件没有从最左列开始连续使用，就可能无法充分利用索引。它的本质是有序索引的匹配规律。

**为什么大厂会问这个**

因为很多索引设计问题，本质都出在没有理解联合索引的使用方式。

**工程要点 / 高频坑点**

不要背成口诀后就结束。更有价值的是结合 where、order by、group by 去解释。

**可勾连项目**

可不强绑项目。

## 面试题95：索引失效的常见场景有哪些？

**标准回答**

常见有函数计算、类型隐式转换、前导模糊查询、不满足最左前缀、低选择性条件和不合理排序。索引不是建了就一定能用，查询写法同样重要。

**为什么大厂会问这个**

这题能看出你是否真正做过 SQL 优化。

**工程要点 / 高频坑点**

不要只记几个例子。主线应该是：查询条件一旦破坏了索引的有序性或可匹配性，就可能导致失效。

**可勾连项目**

可不强绑项目。

## 面试题96：Explain 最该看哪些字段？

**标准回答**

通常优先看 type、key、rows、filtered、extra。它们能帮助你判断是否走了索引、扫描了多少行、是否发生回表或临时表等问题。

**为什么大厂会问这个**

因为看执行计划是后端排查 SQL 性能的基本能力。

**工程要点 / 高频坑点**

不要只说“看有没有用索引”。很多时候用了索引但仍然很慢，rows 和 extra 同样关键。

**可勾连项目**

可不强绑项目。

## 面试题97：事务的 ACID 最该怎么讲？

**标准回答**

原子性保证一组操作要么都成功要么都失败，一致性保证事务前后业务约束不被破坏，隔离性保证并发事务相互影响可控，持久性保证提交后结果不会丢。面试里更重要的是把这些概念和实际设计联系起来。

**为什么大厂会问这个**

这是事务理解的基础题，很多并发和一致性题都会往这里收。

**工程要点 / 高频坑点**

不要只背四个名词。更好的回答是举出数据库、锁和日志各自承担了什么作用。

**可勾连项目**

可轻提 `AISmartQA` 的事务边界。

## 面试题98：MySQL 的隔离级别怎么选？

**标准回答**

要看你更担心脏读、不可重复读还是幻读，以及吞吐量和锁冲突能接受到什么程度。工程里常见默认是可重复读或读已提交，但选择要和业务一致性要求匹配。

**为什么大厂会问这个**

因为隔离级别是并发一致性最常见的入口题。

**工程要点 / 高频坑点**

不要机械背四个级别。更关键的是解释每个级别带来的并发成本和可见性差异。

**可勾连项目**

可不强绑项目。

## 面试题99：MVCC 到底解决什么问题？

**标准回答**

MVCC 解决的是读写并发下，如何让读操作少加锁或不加锁仍能看到一致视图。它通过版本链和快照读，在性能和一致性之间做了平衡。

**为什么大厂会问这个**

这是隔离级别和并发控制的高频追问点。

**工程要点 / 高频坑点**

不要把 MVCC 讲成“万能不加锁方案”。当前读和某些更新场景仍然需要锁。

**可勾连项目**

可不强绑项目。

## 面试题100：行锁、表锁、间隙锁分别是什么？

**标准回答**

行锁锁定具体记录，表锁锁定整张表的访问，间隙锁锁定索引区间以避免某些并发插入带来的幻读问题。它们本质上是在不同粒度上权衡并发和一致性。

**为什么大厂会问这个**

因为理解锁粒度，才能理解为什么有时事务会慢、有时会阻塞。

**工程要点 / 高频坑点**

不要把“用了行锁”理解成一定只锁一行。是否命中索引、扫描范围和隔离级别都会影响实际锁范围。

**可勾连项目**

可不强绑项目。

## 面试题101：死锁是怎么产生的？怎么排查？

**标准回答**

死锁通常是多个事务以不同顺序持有并等待对方资源造成的。排查时要看锁等待、执行顺序、SQL 范围和是否存在交叉更新。

**为什么大厂会问这个**

死锁是并发数据库系统里的真实生产问题，不是纯八股。

**工程要点 / 高频坑点**

更稳的预防手段是统一访问顺序、缩小事务范围、走稳定索引和尽量减少长事务，而不是只依赖数据库自动回滚。

**可勾连项目**

可不强绑项目。

## 面试题102：乐观锁和悲观锁怎么选？

**标准回答**

冲突少、读多写少时更适合乐观锁；冲突高、必须强互斥时更适合悲观锁。核心不是理论分类，而是冲突概率和失败重试成本。

**为什么大厂会问这个**

因为很多业务场景都会面临并发更新和库存式问题。

**工程要点 / 高频坑点**

不要回答成“乐观锁性能一定更高”。高冲突场景下，重试风暴也会很贵。

**可勾连项目**

可不强绑项目。

## 面试题103：主从复制和读写分离，面试里最该讲什么？

**标准回答**

主从复制解决扩展和容灾，读写分离解决读压力，但它们会引入复制延迟和读一致性问题。更稳的答法是既讲收益，也讲主从延迟带来的代价。

**为什么大厂会问这个**

这是数据库扩展性的经典题。

**工程要点 / 高频坑点**

不要把“有从库”理解成天然无限扩容。写压力、复制链路和一致性要求都会限制方案效果。

**可勾连项目**

可不强绑项目。

## 面试题104：什么时候应该考虑分库分表？

**标准回答**

当单库单表在容量、吞吐、维护窗口或热点上已经明显成为瓶颈，且通过索引优化、冷热分层、归档和缓存仍无法解决时，才考虑分库分表。它不是升级版优化，而是架构重构。

**为什么大厂会问这个**

因为他们想看你是不是把复杂方案当成默认最优解。

**工程要点 / 高频坑点**

不要把分库分表讲得太轻松。分布式事务、跨分片查询、全局 ID 和扩容迁移都会显著增加复杂度。

**可勾连项目**

可不强绑项目。

## 面试题105：单列索引和联合索引怎么选？

**标准回答**

要根据最常见查询模式、过滤条件组合和排序需求来选。联合索引通常更贴近真实查询，但不是堆得越多越好。

**为什么大厂会问这个**

因为这是索引设计能力的高频题。

**工程要点 / 高频坑点**

不要“每个字段都建一个索引”。索引设计要围绕查询模式，而不是围绕字段列表。

**可勾连项目**

可不强绑项目。

## 面试题106：为什么深分页会慢？怎么优化？

**标准回答**

因为深分页通常意味着扫描并丢弃大量前置结果，排序和回表成本都很高。常见优化思路是基于游标、覆盖索引、延迟关联或按主键范围翻页。

**为什么大厂会问这个**

这是真实系统里很常见的性能问题。

**工程要点 / 高频坑点**

不要只会说“limit 很大所以慢”。更关键的是要解释为什么扫描和排序成本会上去。

**可勾连项目**

可不强绑项目。

## 面试题107：唯一约束和幂等表设计在后端里为什么重要？

**标准回答**

因为很多重复请求、重复消息和重试场景最终都要靠数据库层面的唯一约束或幂等记录来兜底。应用层判断不可靠时，数据库约束往往是最后防线。

**为什么大厂会问这个**

这题能把数据库设计和接口幂等、消息幂等连起来。

**工程要点 / 高频坑点**

不要只靠“先查再插”判断唯一性。在并发场景里，约束和事务才更可靠。

**可勾连项目**

`AISmartQA` 的异步任务幂等可轻度勾连。

## 面试题108：为什么 afterCommit 往往比事务内直接发消息更稳？

**标准回答**

因为只有事务提交成功后，消息才应该向下游暴露。否则一旦事务内先发消息、后回滚，就会出现“消息已送达但数据库没落稳”的典型一致性问题。

**为什么大厂会问这个**

这是事务边界和异步一致性的核心交叉题。

**工程要点 / 高频坑点**

## 面试题137：强一致和最终一致该怎么讲？

## 面试题138：CAP / BASE 在面试里最该怎么答？

**标准回答**

CAP 说的是分布式系统在一致性、可用性、分区容错之间的约束关系，BASE 更强调用最终一致和软状态换取可用性。面试里更重要的是说明你知道这些理论如何影响实际系统设计，而不是背字母。

**为什么大厂会问这个**

这是系统设计基础理论题，但很多人会答得非常空。

**工程要点 / 高频坑点**

不要把 CAP 理解成“三选二”这种口号后就结束。更好的回答会落到场景和 tradeoff。

**可勾连项目**

可不强绑项目。

## 面试题139：为什么分布式事务通常不是默认首选？

**标准回答**

因为它实现复杂、性能成本高、故障面大，而且很多业务其实可以通过本地事务 + afterCommit + MQ + 幂等 + 补偿来解决。系统设计里，能不用分布式事务的时候，通常会先用更简单可控的办法。

**为什么大厂会问这个**

因为他们想看你是不是会直接把最复杂的方案当成标准答案。

**工程要点 / 高频坑点**

不要把“不用分布式事务”说成“不重视一致性”。真正稳的做法是明确边界，再用补偿和治理闭环兜住。

**可勾连项目**

优先勾连 `AISmartQA` 的 afterCommit + 最终一致。

## 面试题140：高并发系统的性能瓶颈一般应该怎么拆？

**标准回答**

通常先拆成 CPU、内存、IO、网络、数据库、缓存、队列和下游依赖几个方向，再看是容量瓶颈、锁竞争、长尾延迟还是热点问题。性能优化的第一步不是调参，而是定位。

**为什么大厂会问这个**

因为很多候选人一提优化就只会说“加缓存”。

**工程要点 / 高频坑点**

不要在没有监控和基线时盲目优化。真正有效的优化要先找主瓶颈。

**可勾连项目**

`AISmartQA` 可讲检索链和异步链，`AgentOps` 可讲 runtime 延迟与预算。

# 模块10：Python / FastAPI / AI 服务化

## 面试题141：为什么 AI 应用岗位通常更偏 Python？

**标准回答**

因为模型生态、数据处理库和 Agent / LLM 框架在 Python 上更成熟，原型验证和集成速度也更快。并不是 Python 天生更高级，而是当前生态和迭代效率更匹配 AI 场景。

**为什么大厂会问这个**

他们想看你是否理解语言选择背后的生态原因，而不是语言偏好。

**工程要点 / 高频坑点**

不要把回答变成语言优劣争论。更稳的是讲生态成熟度和职责分工。

**可勾连项目**

优先勾连 `AgentOps` 的 Python runtime。

## 面试题142：为什么 Python 的 async / await 在 AI 应用里很重要？

**标准回答**

因为 AI 应用常常会同时等待模型调用、工具调用、外部 API 和 I/O 操作，异步能提高吞吐和资源利用率。它特别适合“等待远多于计算”的服务链路。

**为什么大厂会问这个**

很多 AI 服务并不是真的 CPU 密集，而是外部调用密集。

**工程要点 / 高频坑点**

不要把异步等同于多线程。异步更适合 I/O 协作，但 CPU 密集型任务仍可能需要别的并发手段。

**可勾连项目**

优先勾连 `AgentOps` 的 FastAPI runtime。

## 面试题143：GIL 最该怎么讲，才不会显得很浅？

**标准回答**

更稳的讲法是：GIL 影响的是同一进程内 Python 字节码并行执行能力，它对 CPU 密集型多线程不友好，但对 I/O 密集型服务影响没那么大。面试里不要把 GIL 讲成“Python 不能并发”。

**为什么大厂会问这个**

因为很多候选人一提 Python 并发就只会背 GIL。

**工程要点 / 高频坑点**

回答时要区分 CPU 密集和 I/O 密集，并说明为什么 async/await 仍然有价值。

**可勾连项目**

可轻提 `AgentOps` runtime 的 I/O 型特点。

## 面试题144：为什么 typing 和 Pydantic 对 AI 服务尤其重要？

**标准回答**

因为 AI 服务天然存在不稳定输入和输出，typing 和 Pydantic 能帮助你把接口、配置和模型结构显式化。它们提升的不只是开发体验，还有校验能力和可维护性。

**为什么大厂会问这个**

这是在看你是不是会把 Python 服务写成“脚本堆叠”。

**工程要点 / 高频坑点**

不要把类型标注当成可有可无。对结构化输出、工具参数和请求模型来说，显式类型非常重要。

**可勾连项目**

优先勾连 `AgentOps` 的 runtime 请求与状态模型。

## 面试题145：FastAPI 的依赖注入最值得讲哪些点？

**标准回答**

它能把认证、配置、数据库连接、客户端实例和公共逻辑从接口函数里抽出来，提高复用和测试性。它的价值不在“写法漂亮”，而在于服务结构更清晰。

**为什么大厂会问这个**

因为他们想看你是不是能把 Python 服务写成工程，而不是脚本。

**工程要点 / 高频坑点**

不要把依赖注入滥用成全局状态塞入口。依赖项仍然要有清晰边界和生命周期。

**可勾连项目**

优先勾连 `AgentOps` 的 FastAPI runtime。

## 面试题146：FastAPI 的请求模型、响应模型和异常处理该怎么讲？

**标准回答**

请求模型负责输入校验，响应模型负责输出约束，异常处理负责把内部错误转成稳定可消费的协议。三者一起，服务才像正式接口而不是临时脚本。

**为什么大厂会问这个**

这题能看出你是否理解“AI 服务也要有正常 API 工程规范”。

**工程要点 / 高频坑点**

不要把异常直接透传成原始报错。稳定协议、错误码和可观测性同样重要。

**可勾连项目**

优先勾连 `AgentOps` 的 runtime API。

## 面试题147：流式 AI 接口应该怎么设计？

**标准回答**

要先明确是单向流式输出还是双向交互，再选择 SSE、分块响应或 WebSocket；同时要考虑取消、超时、错误收口和前端消费方式。流式不是为了炫技，而是为了更好的交互和感知延迟。

**为什么大厂会问这个**

AI 应用里，流式输出已经是高频设计话题。

**工程要点 / 高频坑点**

不要只关注“字一个个出来”。更重要的是错误怎么结束、工具结果怎么插入、前端怎么恢复。

**可勾连项目**

可轻提 `AgentOps` 当前仍是轮询，这属于诚实边界。

## 面试题148：Python 服务和 Java 服务在一个 AI 系统里应该怎么分工？

**标准回答**

通常让 Python 承担模型接入、runtime 和数据处理更自然，让 Java 承担鉴权、记录源、业务接口、事务和治理更稳。关键不是语言，而是职责边界清楚。

**为什么大厂会问这个**

这题能看出你对多语言系统是否有清晰分层意识。

**工程要点 / 高频坑点**

不要把分工讲成“哪个语言更强”。更要强调生态、记录源、一致性和演进成本。

**可勾连项目**

优先勾连 `AgentOps`。

## 面试题149：AI 系统的测试应该怎么分层？

**标准回答**

至少要有单元测试、接口测试、关键链路集成测试、评测集回归和故障路径测试。AI 系统的特殊点在于：除了代码逻辑，还要测试模型行为边界和版本回归。

**为什么大厂会问这个**

很多人会说“我测过接口”，但这远远不够。

**工程要点 / 高频坑点**

不要把测试只理解成 happy path。对 AI 系统来说，异常输入、超时、审批拒绝、工具失败都要覆盖。

**可勾连项目**

`AgentOps` 可讲 runtime/Eval 测试，`AISmartQA` 可讲链路回归。

## 面试题150：Docker 化和部署上线，AI 开发工程师最该讲哪些点？

**标准回答**

最该讲的是环境一致性、依赖收敛、配置外置、镜像分层、健康检查和资源限制。AI 服务常常依赖多种客户端和模型配置，容器化的价值是让部署过程可重复、可迁移。

**为什么大厂会问这个**

因为能跑在本地和能稳定上线是两回事。

**工程要点 / 高频坑点**

不要只说“写个 Dockerfile 就行”。真正上线还要考虑镜像大小、启动时间、配置注入和运行时资源。

**可勾连项目**

可不强绑项目，按通用工程回答即可。
