# AgentOps 高频问答清单

## 一. 项目级别

### 1. 这个项目和 AISmartQA 的区别是什么？
答：AISmartQA 更偏工程化 RAG 问答系统，重点在文档解析、embedding、Milvus、ES、Hybrid 和问答闭环；AgentOps 是在这层能力之上再加一层 Agent 编排，把检索和结构化查询都抽象成工具，同时补会话、Run/Step Trace、MCP、异步 Eval 和平台治理能力，定位更偏企业级 AI 应用平台。

### 2. 为什么这个项目先做本地工具，再做 MCP？
答：因为平台项目要先把执行链做完整，再上协议抽象。V1 先把 `提问 -> 规划 -> 调工具 -> 回答 -> Trace` 做稳；V2 再把工具升级成 MCP Server，这样复杂度可控，演进路径也更清晰。

### 3. 现在这个项目的主链是什么？
答：用户登录后创建会话，发起一次 Agent Run，系统先用 Planner 判断是否需要调用工具，再按顺序执行本地工具或 MCP 远程工具，把工具结果回填上下文，最后生成最终答案，同时把每一步落到 `agent_run_step`。如果要做离线质量验证，还可以发起 Eval Run，后台通过 RabbitMQ 批量跑评测并在 Dashboard 里观察结果。

## 二. 基础知识级别

### 4. 什么是 Agent Run，什么是 Step？
答：Run 表示一次完整执行；Step 表示这次执行中的某一步，例如 PLAN、TOOL_CALL、FINAL_ANSWER。Run 负责聚合总状态和总耗时，Step 负责保存每一步的输入、输出、耗时和异常。

### 5. 为什么工具要抽成统一接口？
答：因为工具系统要支持扩展。如果直接在代码里写死 `if toolName == kb_search`，后面新增工具或切换成 MCP 就会很痛。统一接口后，工具只需要实现 `execute`，Agent 主链不需要知道底层是本地工具还是远程工具。

### 6. 为什么 `sql_query` 必须是白名单模板，而不能直接执行任意 SQL？
答：因为 Agent 场景里直接放开任意 SQL 风险极高，容易出现越权、误删和资源打爆。模板化只读查询能把可控性放在服务端，模型只负责选模板和填参数，不直接掌握数据库破坏能力。

### 7. MCP 第一版到底做了什么？
答：当前已经实现了最小可用的 `initialize`、`tools/list`、`tools/call` 三个接口。主系统可以通过 `McpHttpClient` 走远程工具模式调用 `kb_search` 和 `sql_query`，这说明工具协议化已经真正落地，而不是只写在文档里。

## 三. 多维度多高度

### 8. 为什么说这是 AI 工程平台，而不是单纯调用大模型？
答：因为它已经具备平台化关键要素：用户与会话、Run/Step Trace、工具注册、工具执行链、结构化规划、MCP 协议、异步评测、失败样本分析、限流和缓存。它不是一次模型调用，而是把模型放进一个可治理、可观测、可评测的系统里。

### 9. Eval 为什么要异步化？
答：因为评测天然是批处理场景，一次 Eval Run 可能包含多条 case。如果同步跑，接口延迟会很高，还会和主聊天链路争资源。异步化后，用户发起 Eval 请求可以立即返回，后台慢慢消费 case，并实时累计到 Dashboard，这更符合工程化平台设计。

### 10. 为什么做会话摘要缓存？
答：因为 Agent 会话一长，后续就会遇到上下文膨胀问题。当前先做的是最近 N 条消息的摘要缓存，把摘要缓存成 Redis key，后续无论接更复杂的 Planner、Prompt 压缩还是长上下文优化，都有可复用基础。

### 11. 如果面试官追问“这个项目下一步还能怎么升级”，怎么答？
答：第一是把 MCP 从同进程演示版进一步拆成独立 server；第二是补更真实的 Gemini Planner / function calling；第三是把 Eval 从规则评分升级成更强的模型评分或人工标注闭环。这几步都属于平台增强，而不是推翻重写。
