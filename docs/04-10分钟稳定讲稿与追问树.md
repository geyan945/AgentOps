# AgentOps 10 分钟稳定讲稿与追问树

## 一. 10 分钟稳定讲稿

### 1. 项目定位
AgentOps 现在不是“Java 后端包一层大模型接口”的项目，而是一个 `Java control plane + Python LangGraph runtime + React workspace` 的企业级 Agent 应用。它面向企业知识和数据 Copilot 场景，重点体现多步编排、工具调用、审批恢复、记忆、Trace 和 Eval。

### 2. 为什么要做 2.1 补强
2.0 已经补齐了 LangGraph、前端工作台和 human-in-the-loop，但如果继续深挖，面试官一定会追问两个点：状态能不能持久化恢复，LangGraph 里的模型到底是不是“真的在做规划”。所以 2.1 的核心补强就是：

- Java control plane 新增 `agent_runtime_checkpoint`
- Python runtime 用真实 Gemini 驱动 `supervisor_plan / evidence_reviewer / finalize`
- Eval 不再走旁路脚本，而是走真实 runtime 图执行

### 3. 现在的主执行链

```text
用户提问
 -> Java 创建 session / run
 -> Python runtime 拉上下文
 -> intake_guardrail
 -> load_memory
 -> supervisor_plan(Gemini)
 -> knowledge_researcher / data_analyst
 -> evidence_reviewer(Gemini)
 -> human_approval(按条件)
 -> finalize(Gemini)
 -> Java 回写 step / checkpoint / approval / eval
```

### 4. Java control plane 负责什么
Java 现在固定负责系统记录源：

- JWT、用户隔离、session/message
- run / step / human_task / memory_fact
- MCP 工具平面
- `agent_runtime_checkpoint`
- Eval 和 Dashboard API

这样做的原因是：Agent 运行时可以变，但系统记录源不能散。

### 5. 为什么 checkpoint 放 Java/MySQL
这是 2.1 最值得讲的架构点。之前暂停恢复在 Python 进程内存里，更像 demo。现在 checkpoint 放到 Java control plane，有三个收益：

- 运行时重启后仍可恢复
- resumeToken 和 checkpointVersion 可以做防重放
- 所有 run、step、checkpoint、approval 都在一个一致的审计面里

### 6. 为什么 Gemini 放 Python runtime
因为 LangGraph runtime 才是智能体真正做决策的地方。如果 Gemini 还留在 Java 的旧 Planner/AnswerService，面试官会觉得 LangGraph 只是壳。现在：

- `supervisor_plan` 决定 route 和 pending tasks
- `evidence_reviewer` 决定 finalize / replan / human approval
- `finalize` 负责最终回答和 bounded failure 收口

这就能很清楚地说明：LangGraph 里不是空节点，而是真实模型节点。

### 7. 审批怎么讲
审批触发三类场景：

- 高风险 SQL 宽范围统计或导出
- reviewer 低置信度且证据不足
- KB 和 SQL 结果冲突

MANUAL 模式会写 checkpoint、发 human task、下发 resumeToken；AUTO_APPROVE / AUTO_REJECT 则用于 Eval 自动跑审批分支。

### 8. Eval 为什么更强了
现在 Eval 跑的是和在线聊天同一条 runtime 图，而不是 Java 里的旧 planner/answer 旁路逻辑。评测结果除了 `routeScore / groundingScore / citationScore / finalScore`，还会记录：

- `approvalTriggered / approvalDecision`
- `nodePath`
- `retryCount / latencyMs`

所以它能真实回答“这条 Agent 主链是不是稳定、可解释、可评测”。

### 9. 收尾一句话
我现在会把 AgentOps 定义成：一个以 Java control plane 为系统记录源、以 Python LangGraph runtime 为智能决策层、同时具备 durable checkpoint、real Gemini planning、审批恢复和 runtime-based Eval 的企业级 Agent 应用项目。

## 二. 高频追问

### 1. 为什么不直接用 LangGraph 自带内存 checkpoint？
答：因为我更在意系统一致性。run、step、approval、checkpoint 如果分散在多个存储里，审计和恢复会变复杂。放进 Java/MySQL 后，control plane 仍然是唯一系统记录源。

### 2. 为什么 resume 要校验 token 和 checkpointVersion？
答：为了防止重复审批、旧页面重复提交或 stale resume。resumeToken 解决“是不是这次暂停”，checkpointVersion 解决“是不是这版状态”。

### 3. 你现在的 Gemini 是真的在参与图决策吗？
答：是。至少 `supervisor_plan`、`evidence_reviewer`、`finalize` 三个节点是 Gemini 驱动，mock 只在本地没 key 或 CI 环境下才作为回退。

### 4. 为什么 Eval 要和在线主链共用 runtime？
答：因为只有这样评测结果才有代表性。否则线上是一条链，离线评测是另一条链，分数再高也没意义。

### 5. 这个项目还有哪些边界不是生产终态？
答：第一，checkpoint 现在是 control plane 持久化，但还不是 LangGraph 官方 checkpointer 形态；第二，Gemini 输出虽然要求 JSON，但不是 schema registry 级别治理；第三，前端现在还是轮询，不是流式事件总线。
