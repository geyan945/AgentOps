# AgentOps 10 分钟稳定讲稿与追问树（强化版）

## 一. 10 分钟稳定讲稿强化版

### 1. 先用一句话定项目
AgentOps 是一个面向企业知识库与数据分析场景的多工具智能体平台，不是单纯的问答系统。它在 RAG 能力之上进一步补了会话、Run/Step Trace、工具编排、MCP 协议和异步评测，使模型能力进入一个可治理、可评测、可扩展的平台框架。

### 2. 为什么我要做这个项目
因为我已经有 AISmartQA，如果再做一个普通 RAG 项目，和原项目重叠太高。于是我在 AISmartQA 的检索能力之上做 AgentOps，把知识库检索、文档读取、结构化查询都抽象成工具，再通过 Planner 统一编排，让项目从“知识库问答系统”升级成“企业级智能体平台”。

### 3. 系统架构怎么讲
当前是单体平台版，但已经把未来的平台形态拆得比较清楚：
- Auth：JWT 登录和用户隔离
- Session：会话、消息、摘要缓存
- Agent：Run、Step、Planner、Executor、Answer
- Tool：`kb_search`、`doc_fetch`、`sql_query`
- MCP：`initialize / tools/list / tools/call`
- Eval：数据集、异步 Run、结果聚合、Dashboard、失败样本分析
- Infra：MySQL、Redis、RabbitMQ、Elasticsearch

这里最重要的不是模块多，而是每个模块都对应 Agent 平台里的一个平台能力：安全、上下文、执行链、工具体系、协议层、评测层和基础设施层。

### 4. 主执行链怎么讲
主链就是：

```text
用户提问
 -> Planner 规划
 -> Tool Call
 -> Final Answer
 -> Trace 回放
```

具体落地时：
- 用户先创建一个 session
- 每次提问都会生成一个 run
- run 下会产生若干 step
- step 至少包含 `PLAN`、`TOOL_CALL`、`FINAL_ANSWER`
- 每个 step 都会记录输入、输出、耗时和错误信息

这一步是项目的核心，因为它解决了普通 LLM 应用最缺的一件事：**中间过程可见、可追踪、可复盘**。

### 5. 三个工具怎么讲
我当前平台内置了三类工具：
- `kb_search`：基于 AISmartQA 的知识库检索工具
- `doc_fetch`：按文档 ID 读取文档正文和元信息
- `sql_query`：只读白名单 SQL 模板工具，用于统计和数据分析场景

我没有把工具调用写死在 Agent 主流程里，而是统一抽象成 `ToolExecutor`。这样后续无论工具是本地实现、HTTP 服务，还是 MCP Server，主链都不用重写。

### 6. MCP 第一版怎么讲
MCP 这块我不是只停留在概念层。当前已经落了最小可用版：
- `initialize`
- `tools/list`
- `tools/call`

主系统内部有 `McpHttpClient`，并且我做了 `kb_search_remote` 和 `sql_query_remote` 两个远程工具执行器。也就是说，现在 Agent 已经能真正走“远程工具模式”，而不是只会本地函数调用。当前为了控制环境复杂度，MCP 先做成同进程演示版；后续再拆独立 server，就只是部署形态变化，不是重新设计。

### 7. Eval 怎么讲
Agent 平台如果只有执行，没有评测，就很难工程化。当前我做了：
- Eval 数据集
- Eval Run
- Eval Result
- RabbitMQ 异步评测
- Eval Dashboard
- 失败样本分析

流程是：用户创建评测集后，发起一次 eval run，系统把 case 扔进 RabbitMQ，后台逐条跑 Agent，然后沉淀结果、通过率、平均耗时和失败原因。这样平台既能在线执行，也能离线验证效果。

### 8. Redis 在这里起什么作用
Redis 这里不是只做缓存，我主要用了三类能力：
- 运行限流：控制用户每分钟 run/eval 次数
- 工具短期缓存：减少重复工具调用成本
- 会话摘要缓存：为长会话上下文治理打基础

这里的重点是：我把 Redis 用在了平台治理层，而不是只当一个随手用的 key-value 存储。

### 9. 为什么这项目像平台，不像 Demo
因为它已经具备了平台该有的四层东西：
- 能力层：工具系统
- 编排层：Planner / Answer / Run / Step
- 治理层：限流、缓存、Trace、Eval
- 协议层：MCP 第一版

所以它不是“调了一次模型返回个答案”，而是把模型和工具纳入一个可治理的执行系统。

### 10. 最后一句收尾
如果一句话总结，我会说：

**AgentOps 是一个站在 AISmartQA 之上的企业级智能体平台项目，我重点做了多工具抽象、Run/Step Trace、MCP 第一版和异步 Eval，让 Agent 从“能调用模型”变成“可治理、可扩展、可评测的平台能力”。**

## 二. 面试追问树强化版

### A. 项目定位追问链

#### 1. 这个项目和普通 RAG 系统最大的区别是什么？
答：普通 RAG 系统核心是 `检索 -> 生成`，它解决的是“给模型补知识”；AgentOps 在这之上再加了一层“工具选择和执行编排”，也就是让系统不仅能查知识，还能根据问题决定是否调结构化查询、文档读取或者其他工具。所以它的定位从“问答系统”升级成了“智能体平台”。

#### 2. 为什么不直接在 AISmartQA 里继续加工具，而是单独做 AgentOps？
答：因为 AISmartQA 的边界是知识库问答系统，它的核心是文档接入、检索和回答；如果把会话、工具、Trace、MCP、Eval 全都堆进 AISmartQA，会让职责边界变乱。单独做 AgentOps 后，AISmartQA 作为底层知识能力服务，AgentOps 作为上层编排平台，系统分层更清晰，也更适合简历里体现“平台化能力”。

### B. 工具系统追问链

#### 3. 为什么工具一定要统一抽象，而不是直接 if/else 调 Service？
答：因为平台里最容易失控的就是工具耦合。如果主流程直接写死 `if toolName == kb_search`，后面每新增一个工具都要改 Agent 主链，这会让平台很快失去扩展性。统一 `ToolExecutor` 后，工具系统和 Agent 主链是解耦的，主链只依赖接口，不依赖具体实现。

#### 4. 你这里为什么要保留本地工具和远程工具两套能力？
答：因为这是平台演进路径的一部分。本地工具适合先把链路做稳，远程工具适合后续协议化和独立部署。我现在不是二选一，而是先保留本地工具保证可用，再补 MCP 远程工具验证协议链，这样系统的可演进性更强。

#### 5. `sql_query` 为什么只能做白名单模板？
答：因为 Agent 场景里让模型直接拼任意 SQL 风险很大，容易出现越权、误删、性能打爆。白名单模板把 SQL 的控制权留在服务端，模型只负责选模板和给参数，这样既保留了结构化查询能力，又把风险收住了。

### C. MCP 追问链

#### 6. 你现在说做了 MCP，面试官追问“到底做了什么”怎么答？
答：我当前做的是最小可用 MCP 第一版，支持 `initialize`、`tools/list`、`tools/call` 三个核心接口。主系统有 `McpHttpClient` 负责发请求，远程工具执行器负责把 Agent 工具调用转成 MCP 请求，服务端则返回统一的工具描述和调用结果。这说明我已经把“远程工具协议化”真正接进了主链。

#### 7. 为什么现在不直接拆成独立 MCP Server？
答：因为当前更重要的是把协议和主链打通，而不是先把部署形态做复杂。同进程演示版已经能证明协议、执行器、调用链都成立；后面拆成独立服务，变化主要在部署边界，不是在业务主链上重写一遍。

#### 8. 如果面试官问“这不是真正的分布式 MCP 平台吧”，怎么接？
答：我会承认当前是同进程演示版，这是控制复杂度的选择。但这并不影响它已经具备真正 MCP 化的关键特征：协议接口、客户端调用器、远程工具执行器和可替换的工具边界。换句话说，现在完成的是协议化和调用模式，下一步再做服务拆分。

### D. Trace / 可观测性追问链

#### 9. 为什么要做 Run/Step Trace？
答：因为 Agent 系统最大的调试难点就在“中间过程不可见”。如果只有最终答案，出问题时你根本不知道是 Planner 选错工具了、工具参数错了，还是 Answer 阶段总结错了。Run/Step Trace 把中间过程显式落库，能直接回放每一步输入、输出、耗时和异常，这对调试、评测和面试表达都很重要。

#### 10. Run 和 Step 为什么分开建模？
答：Run 代表一次完整执行，Step 代表执行过程中的一个原子阶段。Run 适合存聚合信息，比如总状态、总耗时、最终答案；Step 适合存阶段明细，比如 PLAN、TOOL_CALL、FINAL_ANSWER。这样既能看总览，也能看细节，职责很清晰。

### E. Eval 追问链

#### 11. 为什么 Eval 要异步化？
答：因为评测是批处理，一次 run 往往包含多条 case。如果同步跑，接口会长时间阻塞，而且会和主聊天链路争资源。异步化后，用户发起请求可以立即返回，后台慢慢消费 case，再把聚合结果同步到 dashboard，这更像平台而不是 demo。

#### 12. 为什么要做失败样本分析？
答：因为只看通过率没法指导优化。失败样本分析能告诉你：是工具选错了，还是关键词没命中，还是回答阶段信息不足。也就是说，平台不仅要给出“分数”，还要给出“哪里出错了”。

#### 13. 你现在的 Eval 打分为什么还是规则打分？
答：因为当前阶段我先追求闭环和可解释性。规则打分能稳定跑通数据集、异步执行、结果聚合和失败样本分析。后续如果要继续增强，可以再升级成模型评分或人工标注闭环。

### F. Redis / 治理追问链

#### 14. 会话摘要缓存为什么有必要？
答：因为会话一长，上下文会越来越重。当前先把最近 N 条消息做摘要并缓存到 Redis，既能减少重复读取消息表，也为后续上下文压缩和多轮记忆能力打基础。它不是一个孤立优化点，而是面向长会话治理的基础能力。

#### 15. 工具结果缓存为什么适合做在 Redis？
答：因为像 `kb_search(query=..., topK=...)`、`doc_fetch(documentId=...)` 这类请求在短时间内重复率很高，用 Redis 做短期缓存可以直接降低重复访问知识库和数据库的成本，而且缓存 key 可以天然按工具维度设计，边界很清楚。

#### 16. 限流为什么做到 AgentOps 层，而不是交给网关？
答：因为这里的限流不是纯 HTTP 流量限流，而是平台能力限流，比如“每个用户每分钟最多发多少次 run / eval”。这类限流语义和业务强相关，放在平台服务层更容易根据用户、场景和接口类型做细粒度控制。

### G. 架构与取舍追问链

#### 17. 这个项目最核心的架构取舍是什么？
答：最核心的取舍是“先做可跑的单体平台版，再逐步协议化和远程化”。我没有一开始就拆一堆服务，而是先把本地工具、Trace、Eval 跑通，再逐步补 MCP 和治理能力。这样做的好处是链路稳定、调试成本低、演进路径清晰。

#### 18. 如果面试官问“为什么不用 function calling 一步到位”，怎么答？
答：因为当前我更重视平台可控性。结构化 Planner + ToolExecutor + Step Trace 这套设计，更适合 Java 后端做落库、回放、评测和扩展。function calling 当然可以继续接，但那是能力增强，不是平台主骨架。

#### 19. 这个项目现在最像大厂的点是什么？
答：不是用了多少 AI 名词，而是它已经具备平台化系统常见的几层能力：用户隔离、执行链追踪、工具协议化、异步评测、失败样本分析、Redis 限流和缓存。这些组合起来，已经很像一套真正的企业级 AI 应用平台雏形。

### H. 与其他项目对比追问链

#### 20. AgentOps 和 AISmartQA / DianPing 的互补关系怎么讲？
答：AISmartQA 代表工程化 RAG 检索问答系统，重点在检索和知识处理；DianPing 代表高并发 Redis 工程项目，重点在缓存、Lua、MQ、Feed；AgentOps 则代表 AI 平台化能力，重点在工具编排、协议化、Trace 和 Eval。三个项目组合起来，分别覆盖 AI 检索、传统后端高并发和智能体平台三条线。

#### 21. AgentOps 之后还能怎么继续做强？
答：第一，把 MCP 从同进程演示版拆成独立服务；第二，把 Planner / Answer 升级成更强的 Gemini / function calling；第三，把 Eval 升级成更强的评分和看板；第四，把 prompt 版本管理、工具权限控制和前端 dashboard 做出来。也就是说，现在主链已经闭环，后面做的是平台增强，不是从头返工。
