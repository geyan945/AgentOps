# AgentOps 10 分钟稳定讲稿与追问树（强化版）

## 一. 强化版讲稿

### 1. 一句话先定性
AgentOps 是一个企业知识与数据 Copilot 项目，但它的核心卖点不是“会调模型”，而是 `durable checkpoint + checkpoint replay + Gemini-driven LangGraph runtime + runtime-based Eval + event stream` 这几件事。

### 2. 现在的三层结构

- Java：control plane，负责鉴权、会话、run/step、checkpoint、approval、eval、MCP
- Python：runtime，负责 LangGraph 状态图、多步工具编排、Gemini 决策、恢复执行、skills / team-graph
- React：workspace，负责 trace、graph、live event stream、审批和 dashboard 演示

这三层分工之后，项目的工程边界就非常清楚。

### 3. 面试官最关心的改造点

#### A. checkpoint 从内存改成持久化
旧版本暂停态在 Python 内存里，重启就丢。现在 checkpoint 存在 Java/MySQL：

- 每个节点执行后都会落 checkpoint
- 进入 `human_approval` 前一定先写 checkpoint
- resume 必须带 `resumeToken + checkpointVersion`

这让项目从“能跑”升级成“可恢复、可审计”。

#### A2. replay 从审批态恢复扩展到任意 checkpoint continue

- 审批恢复仍然通过 `resumeToken + checkpointVersion`
- 非审批态现在也能从最新 checkpoint 做 replay / continue
- 这意味着项目不只支持 HITL resume，还支持 crash / continue 风格恢复

#### B. Gemini 从 Java 边缘逻辑改成 runtime 主节点
旧版本的 Gemini 主要在 Java Planner/AnswerService，面试时很容易被质疑“LangGraph 只是包装”。现在真实模型节点在 Python runtime：

- `supervisor_plan`：规划 route 和 pending tasks
- `evidence_reviewer`：判定 grounded / replan / human approval
- `finalize`：生成最终答案或 bounded failure

#### C. Eval 从旁路脚本改成真实主链评测
现在 Eval case 会创建真实 run，直接调 runtime 图执行，而不是走旧的 Java planner/answer 旁路链。这样评测结果和线上执行一致。

#### D. 编排模式从单图扩到 team-graph

- 默认是 `SINGLE_GRAPH`
- 可选 `TEAM_GRAPH`
- TEAM_GRAPH 不是开放式 swarm，而是受控 multi-role / multi-skill 编排

### 4. 现在怎么讲审批
审批不是一个“为了好看硬加的页面”，而是 runtime 的正式分支。高风险 SQL、低置信 reviewer 和证据冲突都会进入 `human_approval`。对于 Eval，我再加了 `AUTO_APPROVE / AUTO_REJECT`，这样审批分支也能自动回归测试。

### 4.1 现在怎么讲事件流

以前我会诚实说前端是轮询。现在我可以说：

- Java control plane 维护 `agent_run_event`
- 前端通过 SSE 订阅 run 事件
- 轮询仍然保留，用于兼容回退

### 5. 现在怎么讲 Eval 指标
我会把评分拆成两段：

- 规则层：检查 route、approval、citation、artifact、retry bound
- Gemini judge：判断 grounding

最后得到 `routeScore / groundingScore / citationScore / finalScore`，同时保留 `approvalTriggered / approvalDecision / nodePath / retryCount / latencyMs`。

### 6. 最强收尾
如果面试官问“这个项目最像大厂的地方是什么”，我会回答：不是用了多少 AI 名词，而是我把智能体运行时、系统记录源、事件流和质量评测闭环都拆清楚了，而且 checkpoint、审批恢复、checkpoint replay 和 Eval 跑的是同一条主链。

## 二. 追问树强化版

### 1. 为什么 checkpoint 不放 Redis？
答：Redis 更快，但会让运行态脱离 control plane。我的目标是把 run、step、approval、checkpoint 全部沉淀在 Java/MySQL 一致性视图里，面试叙事更完整。

### 2. 为什么 Gemini 要放 Python runtime，而不是继续放 Java？
答：因为 Python runtime 才是 LangGraph 图执行层。只有把规划、review、finalize 放进去，LangGraph 才不是“展示图”，而是“真正做推理和决策的层”。

### 3. 你现在是不是多了一套旧 Java planner/answer？
答：主链和 Eval 主链都已经切到 runtime 了。旧 Java planner/answer 只保留作 legacy fallback，不再是核心执行路径。

### 4. 为什么要保留 mock 模式？
答：因为本地 CI 和无 key 环境需要稳定可跑。生产或面试演示时给 `GEMINI_API_KEY` 就走真实 Gemini，没有 key 时自动退回 mock，保证工程可测试性。

### 5. reviewer 和 supervisor 的分工是什么？
答：supervisor 负责“下一步做什么”，reviewer 负责“当前证据够不够、要不要重规划、要不要审批”。把这两层拆开后，多步执行链更容易讲清楚。

### 6. 为什么 Eval 要记录 nodePath？
答：因为 Agent 项目不是只有最终答案是否正确，更重要的是“是不是按正确路径执行”。nodePath 能回答工具有没有走错、审批有没有触发、replan 有没有发生。

### 7. 这个项目最大的技术 tradeoff 是什么？
答：我优先选择了“Java 统一记录 + Python 专注 runtime”的清晰分层，而不是把所有能力都堆在一个语言里。这样协调成本更高一些，但项目表达力和工程边界都更强。

### 8. 现在还缺什么才算更接近生产？
答：更完整的 schema-governed prompt 管理、真正的官方 durable checkpointer 兼容、以及更强的权限和规模化运维能力。事件流 UI 已经补上，但它仍然只是平台能力的一部分，不改变主链边界。
